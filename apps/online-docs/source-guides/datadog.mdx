# Datadog

Access metrics, dashboards, monitors, and infrastructure data from your Datadog account.

<Note>
**Multi-Header Authentication Required**

Datadog requires two authentication headers: `DD-API-KEY` and `DD-APPLICATION-KEY`. Get both from [Datadog Organization Settings](https://app.datadoghq.com/organization-settings/api-keys).
</Note>

## API Reference

This source provides a single flexible `api_datadog` tool that accepts:
- `path`: API endpoint (e.g., "api/v1/query")
- `method`: HTTP method (GET, POST, etc.)
- `params`: Request body or query parameters

### Common Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| api/v1/query | GET | Query time-series metrics |
| api/v1/metrics | GET | List available metrics |
| api/v1/dashboard | GET | List all dashboards |
| api/v1/dashboard/{id} | GET | Get dashboard details |
| api/v1/monitor | GET | List all monitors |
| api/v1/monitor/{id} | GET | Get monitor details |
| api/v1/hosts | GET | List hosts |
| api/v1/validate | GET | Validate API credentials |

## Metric Query Syntax

Query metrics using Datadog's query language:
- `avg:system.cpu.user{*}` - Average CPU usage across all hosts
- `sum:requests.count{service:web}` - Total requests for web service
- `max:memory.used{host:server1}` - Max memory on specific host

**Time parameters** (Unix timestamps):
- `from`: Start time
- `to`: End time

Example: `api/v1/query?query=avg:system.cpu.user{*}&from=1704067200&to=1704153600`

## Guidelines

- **Rate limits**: Datadog has per-endpoint rate limits. Avoid excessive polling.
- **Time ranges**: Use reasonable time windows (hours/days, not months) for metric queries.
- **Metric names**: Use exact metric name format (e.g., `system.cpu.user`, `aws.ec2.cpuutilization`).
- **Read-only**: This is typically configured for read-only access to monitoring data.

## Setup Guide

### Configuration

**Required config.json:**
```json
{
  "name": "Datadog",
  "slug": "datadog",
  "enabled": true,
  "provider": "datadog",
  "type": "api",
  "api": {
    "baseUrl": "https://api.datadoghq.com/",
    "authType": "header",
    "headerNames": ["DD-API-KEY", "DD-APPLICATION-KEY"],
    "testEndpoint": {
      "method": "GET",
      "path": "api/v1/validate"
    }
  }
}
```

<Note>
**Regional URLs**

The base URL varies by Datadog region:
- US1 (default): `https://api.datadoghq.com/`
- US3: `https://api.us3.datadoghq.com/`
- US5: `https://api.us5.datadoghq.com/`
- EU: `https://api.datadoghq.eu/`
- AP1: `https://api.ap1.datadoghq.com/`
</Note>

### Getting API Credentials

1. Log into [Datadog](https://app.datadoghq.com)
2. Go to **Organization Settings** → **API Keys**
3. Create or copy an existing API Key
4. Go to **Organization Settings** → **Application Keys**
5. Create or copy an existing Application Key

<Warning>
Both keys are required. The API Key authenticates your organization, while the Application Key authenticates you as a user with specific permissions.
</Warning>

### Authentication

After creating the source, use:
```
source_credential_prompt({
  sourceSlug: "datadog",
  mode: "header",
  labels: {
    "DD-API-KEY": "API Key",
    "DD-APPLICATION-KEY": "Application Key"
  },
  description: "Enter your Datadog API credentials",
  hint: "Find these in Organization Settings → API Keys / Application Keys"
})
```

### Recommended Questions

- Which Datadog region are you using? (US1, EU, etc.)
- What do you primarily want to monitor? (metrics, dashboards, alerts)
- Are there specific services, hosts, or tags you focus on?
- Do you need read-only access or will you need to create monitors/dashboards?

### Common Use Cases

**Query CPU usage for the last hour:**
```
GET api/v1/query?query=avg:system.cpu.user{*}&from={unix_now-3600}&to={unix_now}
```

**List monitors in alert state:**
```
GET api/v1/monitor?monitor_tags=status:alert
```

**Get dashboard list:**
```
GET api/v1/dashboard
```

**Validate credentials:**
```
GET api/v1/validate
```

### Best Practices

- **Use specific time ranges** to limit data volume
- **Use tags for filtering** (`host:server1`, `service:web`, `env:production`)
- **Cache dashboard IDs** in guide.md if the user works with specific dashboards
- **Monitor rate limits** to avoid hitting quotas
- **Prefer aggregate queries** (`avg:`, `sum:`, `max:`) over raw data for efficiency

### Rate Limits

- Rate limits vary by endpoint and Datadog plan
- Most read operations have generous limits
- Avoid rapid sequential requests to the same endpoint
- Check [Datadog Rate Limits](https://docs.datadoghq.com/api/latest/rate-limits/) for specifics
