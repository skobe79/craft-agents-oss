# Owner Agent — Temporary Product Identity

> Working identity only. Replace the name after the owner chooses the final product name; preserve the underlying product principles.

## Product idea

**Owner Agent** is a private command environment for one person. It is not a team chat product, a customer-support bot, or a generic AI wrapper. It should feel like a powerful local workstation that happens to converse—not a website with a chat box.

## Product promise

**Say what you want. See it happen. Keep the result.**

The interface makes five things obvious at all times:

1. what the owner asked;
2. which model/runtime is responsible;
3. what capabilities and filesystem scope are active;
4. what is running, blocked, or complete;
5. where outputs and changes landed.

## Personality

- decisive rather than theatrical;
- technical without becoming clinical;
- direct rather than deferential;
- calm under failure;
- transparent about real limitations;
- never padded with fake enthusiasm or corporate reassurance.

## Naming rules

- “Owner Agent” is the temporary internal codename.
- Do not introduce “AI”, “GPT”, “Craft”, “Copilot”, or “Assistant” into the final trademark without an explicit naming decision.
- Feature names should describe jobs: Command, Runs, Projects, Memory, Media Lab, Integrations.
- Avoid fantasy jargon for ordinary controls.

## Visual thesis: dark operational workstation

The primary experience is a dense but readable dark command environment:

- near-black graphite canvas, not neutral SaaS grey;
- layered charcoal surfaces separated by luminance rather than excessive borders;
- electric cyan as the owner/action signal;
- ember orange for attention and queued decisions;
- green only for verified success;
- red only for failure/destructive scope;
- compact typography with generous line-height in generated content;
- monospaced metadata used selectively for paths, models, durations, IDs, and tool output;
- motion communicates state transitions—queued → running → verified—not decoration.

## Layout model

### Desktop

1. **Activity rail** — narrow fixed rail for Command, Runs, Projects, Memory, Media Lab, Integrations and Settings.
2. **Context pane** — sessions, project tree, run list or integration list depending on destination.
3. **Command stage** — conversation and primary work surface.
4. **Inspector** — optional right pane for run details, artifacts, diffs, memory provenance or generation parameters.

The command stage remains visually dominant. Context and Inspector can collapse independently.

### Compact

- activity rail becomes a bottom or compact side destination bar;
- context pane becomes a slide-over route;
- command stage remains the default landing surface;
- inspector becomes a full-height detail route.

## Core surfaces

### Command

- conversational timeline;
- clear active runtime/model capsule;
- working-directory and policy scope;
- capability-bundle selector;
- attachments/media;
- queue/steer control;
- stop button visible during execution;
- tool runs collapsed by default but never hidden.

### Runs

- foreground and background execution in one timeline;
- parent/child delegation tree;
- status, duration, model, spend/tokens when available;
- transcript and artifact links;
- pause, cancel, retry and duplicate where supported.

### Memory

- profile, durable facts, project memories, session episodes and skills;
- provenance and confidence visible;
- edit, pin, merge, forget and export;
- no secret values permitted.

### Media Lab

- workflow browser and typed parameter forms;
- live ComfyUI queue/progress;
- generated image/video/audio gallery;
- workflow, model, seed and input provenance;
- send output back into Command.

## Brand removal criteria

Before shipping:

- no Craft logo, icon, wordmark or product title remains in the executable UI;
- package/application identifiers are renamed deliberately;
- Apache-2.0 license and required notices remain;
- upstream project is credited in third-party notices, not presented as the new product identity.
